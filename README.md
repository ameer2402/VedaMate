Conversational AI Teaching Assistant
This project is a sophisticated, conversational AI assistant built with Python, Streamlit, and Google's Generative AI models. It uses a Retrieval-Augmented Generation (RAG) architecture powered by a multi-agent system built with LangGraph.
The application allows users to upload their own PDF documents, which are then processed and stored in a vector database. The user can then "chat" with their document, asking questions and getting detailed, context-aware answers. The assistant can also perform live web searches to supplement its knowledge and provide the most current information.
Key Features
PDF Document Chat: Upload any PDF and ask questions about its content.
Multi-Agent Workflow: Utilizes a sophisticated agentic system (built with LangGraph) to intelligently handle user queries:
Query Rewriter Agent: Rewrites conversational follow-up questions into standalone queries for better search results.
Multi-Query Agent: Generates multiple diverse search queries from a single user question to retrieve more comprehensive context.
Router Agent: Classifies the user's intent (e.g., factual question, reasoning, greeting) to direct the query down the most appropriate path.
And more agents for specific tasks like web search, database querying, and final response generation.
Hybrid Search: Retrieves context from both the user's uploaded PDF (via a ChromaDB vector store) and from the live web (via Google's Custom Search API).
Context-Aware Responses: The final answer is generated by a Gemini model using the combined context from the PDF and web search, ensuring a detailed and accurate response.
Interactive UI: A clean and user-friendly web interface built with Streamlit.
Architecture & Tech Stack
The application follows a modern RAG pattern orchestrated by a LangGraph state machine.
User Query: The user asks a question through the Streamlit UI.
Agentic Workflow (LangGraph): The query enters a graph where multiple "agents" (nodes) process it sequentially and in parallel:
The query is rewritten and expanded.
The user's intent is classified.
Based on the intent, the system performs parallel searches on the PDF vector database (ChromaDB) and the web.
The retrieved context is gathered.
Response Generation (Gemini): The final agent passes the original query, chat history, and all retrieved context to a Google Gemini model, which generates the final, formatted answer.
UI Update: The answer is streamed back to the Streamlit interface.
Frontend: Streamlit
Backend/Workflow: LangGraph
AI Model: Google Gemini (gemini-2.0-flash)
Embeddings: Google Generative AI Embeddings (text-embedding-004)
Vector Database: ChromaDB
Web Search: Google Custom Search API

code
Bash

5. Configure Your Environment (.env file)
This is the most critical setup step. You must provide your own Google Cloud credentials.
A. Create the .env File
In the root directory of the project, create a new file named exactly .env.
B. Add Content to the .env File
Copy the following template into your new .env file.
code
Env
# .env file

# Key for Google AI services (Gemini, Embeddings) and Google Search
GOOGLE_API_KEY="YOUR_GOOGLE_API_KEY_HERE"

# Unique ID for your Programmable Search Engine
GOOGLE_CSE_ID="YOUR_GOOGLE_CSE_ID_HERE"
C. Obtain Your Credentials
Note: It is highly recommended to use a personal Google Account (e.g., @gmail.com) to create a new Google Cloud project. Corporate or school accounts often have security policies that can block the required APIs.
Create a Google Cloud Project & Get API Key:
Go to the Google Cloud Console (use an Incognito window to ensure you can log in with a personal account).
Create a New Project.
Enable APIs: In your new project, search for and ENABLE these two APIs:
Generative Language API
Custom Search API
Create API Key: Navigate to APIs & Services > Credentials, click + CREATE CREDENTIALS, select API key, and copy the generated key. Paste it as the value for GOOGLE_API_KEY.
Get Your Programmable Search Engine ID:
Go to the Programmable Search Engine control panel.
Click Add to create a new search engine.
In the setup, turn ON the option to "Search the entire web".
After creation, copy the "Search engine ID" from the control panel and paste it as the value for GOOGLE_CSE_ID.
6. Running the Application
Once your .env file is configured, you can start the Streamlit application.
code
Bash
streamlit run app.py```

Your browser should automatically open a new tab with the application running.

A user uploads their document, and the AI assistant can answer questions using knowledge derived both from the document itself and from real-time web searches, providing a truly comprehensive and personalized learning experience.
How It Works: The Technical Architecture & Workflow
Our application is built on a sophisticated Retrieval-Augmented Generation (RAG) architecture powered by a multi-agent system using LangGraph. This isn't just a simple RAG pipeline; it's an intelligent, decision-making workflow.
Step 1: Document Ingestion: The user uploads a PDF. We chunk the document, generate embeddings using Google's text-embedding-004 model, and store it in a ChromaDB vector database. This creates a searchable knowledge base of the document.
Step 2: The Multi-Agent Workflow Begins: When a user asks a question, it triggers our LangGraph workflow.
Query Rewriter Agent: If the user asks a follow-up question (e.g., "why is that?"), this agent intelligently rewrites it into a complete, standalone question using the chat history for context.
Multi-Query Agent: To ensure we don't miss any relevant information, this agent expands the single user query into 5-6 diverse, parallel search queries.
Router Agent: This is the "brain" of our system. It analyzes the user's intent and classifies the query (e.g., a factual lookup, a request for a summary, a simple greeting). This decision dictates the path the workflow will take.
Step 3: Hybrid & Parallel Search (The Optimization):
If the router decides it's a factual query, it triggers two searches simultaneously using Python's asyncio:
Vector DB Agent: Searches the user's PDF content in ChromaDB.
Web Search Agent: Searches the live web using the Google Custom Search API.
This parallel processing drastically reduces latency compared to a sequential approach.
Step 4: Response Generation: The combined, de-duplicated context from both the PDF and the web is passed to our final agent. This agent uses Google's gemini-2.0-flash model to synthesize all the information and generate a single, coherent, and accurate answer, complete with citations from the PDF.
Why We Are Better Than Existing Approaches:
vs. Standard PDF Readers: We offer semantic, conversational search, not just keyword matching. We find concepts, not just words.
vs. Generic Chatbots: Our answers are grounded in the user's actual course material, eliminating hallucinations and ensuring relevance. We provide the best of both worlds: specific knowledge from the text and broad context from the web.
vs. Simple RAG Systems: Our multi-agent workflow is more robust. The router agent prevents unnecessary web searches for simple greetings, and the multi-query agent provides a richer, more comprehensive context, leading to superior answer quality.
